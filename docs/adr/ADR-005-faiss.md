# ADR-005: FAISS를 벡터 저장소로 선택

## Status
Accepted (2026-03-10)

## Context
- voyage-finance-2로 생성한 임베딩 벡터를 저장하고 유사도 검색을 수행할 벡터 저장소 필요
- 외부 서비스 의존 없이 로컬에서 완결되는 구조 선호 (Docker Compose 단위 배포 원칙)
- 현재 인덱스 규모: 약 5,777 chunks (소규모)

## Decision
faiss-cpu를 로컬 벡터 인덱스로 사용.

고려한 대안:
- **Chroma**: Python 네이티브, 영속성 내장이나 소규모 프로젝트에서 FAISS 대비 성능 이점 없음
- **Pinecone**: 관리형 서비스로 편리하나, 외부 API 의존 + 유료 과금
- **Qdrant**: 고급 필터링 지원이나, 별도 서비스 운용 필요

선택 근거: faiss-cpu는 Meta가 개발한 검증된 벡터 검색 라이브러리로, 5,000~10,000 규모의 인덱스에서 밀리초 단위 검색이 가능하다. 외부 서비스 없이 MCP 서버 컨테이너 내에서 직접 로드하여 운용한다. Docker Compose의 volumes 마운트(`./data/faiss:/app/data/faiss:ro`)로 빌드 시점에 생성한 인덱스를 런타임에 읽기 전용으로 제공한다.

## Consequences
- 긍정: 외부 서비스 의존 없음, 밀리초 단위 검색 속도, 검증된 라이브러리
- 부정: 인메모리 기반으로 프로세스 재시작 시 .index 파일에서 재로드 필요 (휘발성). 대규모 인덱스(수백만 건) 시 메모리 부담
- 볼륨 마운트 전략: `data/faiss/` 디렉토리를 Docker volume으로 마운트하여 인덱스 영속화. 인덱스 갱신은 오프라인 스크립트(scripts/)로 수행 후 컨테이너 재시작
