# RAG 평가 사이클 구현 비효율 사례

> **status**: current
> **작성일**: 2026-03-11
> **대상 세션**: Phase 1-1 RAGAS 평가 및 검색 품질 개선

---

## 개요

RAGAS 평가에서 `context_precision 0.0` 문제를 해결하는 과정에서 불필요한 반복 작업이 발생했다.
총 3회의 재인덱싱과 2회의 RAGAS 평가가 수행되었으나, 1회의 진단 + 1회의 재인덱싱 + 1회의 재평가로 충분했다.

---

## 비효율 발생 경위

### Round 1 — 보일러플레이트 마커 필터 추가
- **진단**: RAGAS 결과 `context_precision 0.0` → 보일러플레이트(표지/목차) 청크가 상위 랭크 점유
- **조치**: `_BOILERPLATE_MARKERS` 기반 필터 추가 후 재인덱싱 (10,286 → 10,286 chunks, 보일러플레이트 0)
- **결과**: RAGAS 재평가 → `context_precision` 여전히 0.0, `faithfulness` 0.817→0.553 하락
- **놓친 것**: 파싱 버그 미발견, 자회사 목록 청크 문제 미파악

### Round 2 — 테이블형 청크 필터 추가
- **진단**: 검색 결과 자회사 목록이 상위 랭크 → 테이블형 데이터 필터 추가
- **조치**: `avg_line_len < 20` 필터 추가 후 재인덱싱 (10,286 → 1,647 chunks)
- **결과**: 검색 결과 확인 중 자회사 목록 일부 여전히 통과 확인
- **놓친 것**: 법인명 패턴(Co., Ltd. 등) 청크가 다른 경로로 통과 중

### Round 3 — 법인명 패턴 필터 추가 + 파싱 버그 수정
- **진단**: `rag_search_client.py` raw 출력 확인 → 멀티라인 내용 블록의 첫 줄만 추출하는 버그 발견
- **조치**: 파싱 버그 수정 + 법인명 필터 추가 후 재인덱싱 (1,647 → 1,496 chunks)
- **결과**: 커밋 완료. RAGAS 재평가는 다음 세션으로 연기

## 핵심 원인 — 진단 순서의 역전

```
실제 진행 순서:
  증상 발견 → 가설 수립 → 조치 → 재평가 → 증상 지속 → 반복

올바른 순서:
  증상 발견 → raw 데이터 확인(파싱 버그) → 검색 결과 full 텍스트 확인 →
  필터 설계(한번에) → 재인덱싱(1회) → 재평가(1회)
```

**파싱 버그(`rag_search_client.py`)가 context_precision 0.0의 주요 원인이었을 가능성이 높다.**
RAGAS에 전달된 컨텍스트가 500자 청크가 아닌 2~30자 첫 줄 조각이었으므로,
어떤 필터를 추가해도 평가 결과가 개선될 수 없었다.

## 재발 방지 원칙

### 1. 파이프라인 버그는 평가 전에 먼저 검증한다

RAG 평가 실패 시 **평가 파이프라인 자체**를 먼저 의심한다.

```python
# 평가 전 파싱 결과를 반드시 출력해서 확인
contexts = search_documents_local(question, corp_name, year)
for i, ctx in enumerate(contexts):
    print(f"[{i+1}] len={len(ctx)}: {ctx[:100]}")
```

### 2. Raw 출력을 먼저 확인한다

점수가 예상과 다를 때 순서:
1. `rag_search()` raw 출력 확인
2. 파싱 결과(`contexts` 리스트) 확인
3. 그 다음 인덱스/청크 품질 확인

### 3. 필터는 한 번에 설계하고 1회만 재인덱싱한다

- 필터 추가 전 **전체 인덱스 통계 분석**을 먼저 수행
- 여러 필터 유형을 한 번에 설계 후 단일 재인덱싱

```
분석(오프라인) → 설계 → 구현 → 테스트 → 재인덱싱(1회) → 재평가(1회)
```

### 4. 비싼 작업(임베딩, LLM 평가)은 사전 검증 후 실행한다

- 임베딩 전: `_is_boilerplate()` 필터 결과 수 확인 후 임베딩 실행
- RAGAS 실행 전: 샘플 1개로 컨텍스트 품질 수동 확인
